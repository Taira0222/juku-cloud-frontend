## ビジネス用語

### アカウント (Account)
**定義**: サービスを利用するための個人または組織の識別情報と設定の集合体

**関連用語**: ユーザー、プロフィール

**使用例**: 「新規アカウントを作成する」「アカウント設定を変更する」

**備考**: 1人のユーザーが複数のアカウントを持つことも可能

---

### サブスクリプション (Subscription)
**定義**: 定期的な支払いによってサービスを継続利用する契約形態

**英語**: Subscription

**関連用語**: プラン、課金

**プラン種類**:
- Free: 無料プラン
- Basic: 基本プラン (月額980円)
- Pro: プロプラン (月額2,980円)
- Enterprise: エンタープライズプラン (要問い合わせ)

**使用例**: 「サブスクリプションをアップグレードする」

## 技術用語

### JWT (JSON Web Token)
**定義**: JSON形式のデータを安全に送受信するためのトークンベースの認証方式

**フルスペル**: JSON Web Token

**用途**: ユーザー認証、API認可

**構造**:
```
Header.Payload.Signature
```

**関連用語**: 認証、トークン、セッション

**参考**: [RFC 7519](https://tools.ietf.org/html/rfc7519)

---

### SSR (Server-Side Rendering)
**定義**: サーバー側でHTMLを生成してクライアントに返すレンダリング手法

**対義語**: CSR (Client-Side Rendering)

**メリット**:
- 初期表示が高速
- SEOに有利
- JavaScriptが無効でも表示可能

**デメリット**:
- サーバー負荷が高い
- TTI (Time to Interactive) が遅くなる可能性

**使用例**: 「ランディングページはSSRで実装する」

---

### API エンドポイント
**定義**: クライアントがサーバーと通信するための特定のURL

**形式**: `[HTTPメソッド] /api/[バージョン]/[リソース]`

**例**:
- `GET /api/v1/users` - ユーザー一覧取得
- `POST /api/v1/users` - ユーザー作成
- `PUT /api/v1/users/:id` - ユーザー更新
- `DELETE /api/v1/users/:id` - ユーザー削除

**関連用語**: REST API, RESTful

---

## ドメイン用語

### ワークスペース (Workspace)
**定義**: 複数のユーザーが共同作業を行うための仮想的な作業空間

**階層構造**:
```
Organization (組織)
 └── Workspace (ワークスペース)
      └── Project (プロジェクト)
           └── Task (タスク)
```

**権限**:
- Owner: 全ての操作が可能
- Admin: 管理操作が可能
- Member: 閲覧・編集が可能
- Guest: 閲覧のみ可能

**使用例**: 「営業部のワークスペースを作成する」

---

### タスク (Task)
**定義**: プロジェクト内の具体的な作業単位

**ステータス**:
- `todo`: 未着手
- `in_progress`: 進行中
- `review`: レビュー中
- `done`: 完了
- `archived`: アーカイブ済み

**属性**:
- タイトル (必須)
- 説明
- 担当者
- 期限
- 優先度 (High / Medium / Low)
- タグ

**使用例**: 「タスクのステータスを更新する」

---

## 略語

### CRUD
**フルスペル**: Create, Read, Update, Delete

**定義**: データベース操作の基本的な4つの機能

**対応するHTTPメソッド**:
- Create → POST
- Read → GET
- Update → PUT/PATCH
- Delete → DELETE

---

### SPA
**フルスペル**: Single Page Application

**定義**: 単一のHTMLページで動作するWebアプリケーション

**特徴**:
- ページ遷移時にリロードしない
- 高速なUI更新
- リッチなユーザー体験

**例**: React, Vue.js, Angular で構築されたアプリケーション

---

### API
**フルスペル**: Application Programming Interface

**定義**: ソフトウェア間でデータをやり取りするための仕組み

**種類**:
- REST API
- GraphQL API
- gRPC

---

### CI/CD
**フルスペル**: Continuous Integration / Continuous Deployment

**定義**: 継続的インテグレーション・継続的デプロイメント

**CI (Continuous Integration)**:
- コードの自動ビルド
- 自動テスト実行
- コード品質チェック

**CD (Continuous Deployment)**:
- 自動デプロイ
- リリースの自動化

**ツール例**: GitHub Actions, GitLab CI, Jenkins

---

### ORM
**フルスペル**: Object-Relational Mapping

**定義**: オブジェクト指向プログラミング言語でデータベースを操作するための技術

**メリット**:
- SQLを直接書かなくて良い
- 型安全
- データベース移行が容易

**例**: Prisma, TypeORM, Sequelize

---

## UI/UX用語

### モーダル (Modal)
**定義**: 画面上に重ねて表示される対話型のウィンドウ

**種類**:
- 確認ダイアログ
- フォーム入力
- 詳細表示

**特徴**:
- 背景をオーバーレイで覆う
- 閉じるまで他の操作ができない(モーダル)
- ESCキーで閉じられる

**使用例**: 「削除確認モーダルを表示する」

---

### トースト (Toast)
**定義**: 画面端に一時的に表示される通知メッセージ

**用途**:
- 成功メッセージ
- エラー通知
- 情報通知

**特徴**:
- 自動的に消える
- ユーザー操作を妨げない
- 複数同時表示可能

**使用例**: 「保存成功のトーストを表示する」

---

### ブレッドクラム (Breadcrumb)
**定義**: 現在のページ位置を階層的に示すナビゲーション

**形式**: `ホーム > カテゴリ > サブカテゴリ > 現在のページ`

**用途**: ユーザーが現在地を把握し、上位階層に戻りやすくする

---

### レスポンシブデザイン (Responsive Design)
**定義**: 画面サイズに応じてレイアウトを最適化するデザイン手法

**ブレークポイント**:
- Mobile: < 640px
- Tablet: 640px - 1024px
- Desktop: > 1024px

**手法**:
- Fluid Grid (流動的グリッド)
- Flexible Images (柔軟な画像)
- Media Queries (メディアクエリ)

---

## ステータスコード

### HTTPステータスコード

#### 2xx 成功
- **200 OK**: リクエスト成功
- **201 Created**: リソース作成成功
- **204 No Content**: 成功(レスポンスボディなし)

#### 4xx クライアントエラー
- **400 Bad Request**: リクエストが不正
- **401 Unauthorized**: 認証が必要
- **403 Forbidden**: アクセス権限なし
- **404 Not Found**: リソースが見つからない
- **422 Unprocessable Entity**: バリデーションエラー

#### 5xx サーバーエラー
- **500 Internal Server Error**: サーバー内部エラー
- **502 Bad Gateway**: ゲートウェイエラー
- **503 Service Unavailable**: サービス利用不可

## データベース用語

### マイグレーション (Migration)
**定義**: データベーススキーマの変更履歴を管理し、適用する仕組み

**用途**:
- テーブル作成・削除
- カラム追加・変更
- インデックス作成

**ファイル命名**: `YYYYMMDDHHMMSS_description.sql`

**例**: `20240101120000_create_users_table.sql`

---

### シード (Seed)
**定義**: 開発・テスト用の初期データを投入すること

**用途**:
- 開発環境のテストデータ作成
- デモデータの準備

**実行タイミング**: マイグレーション後

---

### トランザクション (Transaction)
**定義**: 複数のデータベース操作を1つの処理単位としてまとめる仕組み

**ACID特性**:
- **A**tomicity (原子性): 全て成功か全て失敗
- **C**onsistency (一貫性): データの整合性を保つ
- **I**solation (独立性): 他の処理と干渉しない
- **D**urability (永続性): コミット後は永続化

---

## 認証・認可用語

### 認証 (Authentication)
**定義**: ユーザーが本人であることを確認すること

**方法**:
- パスワード認証
- 多要素認証 (MFA)
- 生体認証
- SSO (Single Sign-On)

---

### 認可 (Authorization)
**定義**: 認証されたユーザーが特定のリソースにアクセスできるかを判断すること

**方法**:
- RBAC (Role-Based Access Control): ロールベース
- ABAC (Attribute-Based Access Control): 属性ベース

**使用例**: 「管理者ロールのみがユーザー削除を認可される」

---

## パフォーマンス用語

### キャッシュ (Cache)
**定義**: 頻繁にアクセスされるデータを一時的に保存し、高速にアクセスできるようにする仕組み

**種類**:
- **ブラウザキャッシュ**: クライアント側
- **CDNキャッシュ**: エッジサーバー
- **アプリケーションキャッシュ**: Redis, Memcached
- **データベースキャッシュ**: クエリ結果

**戦略**:
- Cache-First
- Network-First
- Stale-While-Revalidate

---

### レイテンシ (Latency)
**定義**: リクエストを送信してからレスポンスが返ってくるまでの遅延時間

**単位**: ミリ秒 (ms)

**目標値**:
- API レスポンス: < 200ms
- ページ表示: < 1000ms

---

### スループット (Throughput)
**定義**: 単位時間あたりに処理できるリクエスト数

**単位**: RPS (Requests Per Second)

**例**: 「このAPIは1000 RPSを処理できる」

---

## セキュリティ用語

### XSS (Cross-Site Scripting)
**定義**: Webアプリケーションに悪意のあるスクリプトを注入する攻撃

**対策**:
- 入力値のサニタイゼーション
- エスケープ処理
- Content Security Policy (CSP)

---

### CSRF (Cross-Site Request Forgery)
**定義**: ユーザーが意図しないリクエストを送信させる攻撃

**対策**:
- CSRFトークン
- SameSite Cookie
- Refererチェック

---

### SQLインジェクション
**定義**: SQLクエリに悪意のあるコードを注入する攻撃

**対策**:
- プレースホルダ使用
- ORM使用
- 入力値バリデーション
